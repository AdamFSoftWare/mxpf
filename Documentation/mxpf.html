Links Changelog

MXPF is an xEdit scripting library that provides a variety of functions for the generation of patches. Want to rebalance armor and weapons in Skyrim? Done. Apply a new particle effect to every weather? Easy. Change the configuration of all NPCs? Cake.

MXPF is comprised a suite of functions that do the work of finding the records you want to patch, identifying/creating a patch file, and copying records to the patch. Where you would normally have to learn to do all of these steps yourself, with MXPF you can accomplish everything you need with a few simple functions.

The project is available on GitHub and Nexus Mods. A test suite is currently being developed. There are also a bunch of example scripts available which show you how you can use MXPF and xEdit Scripting to solve problems.

You can report bugs and discuss features on the GitHub issues page. You can also talk with developers and other users on the xEdit HipChat, in the NexusMods comments section, and on r/xEdit.

Dependencies

  • mteFunctions: Comes packaged with mxpf. Updates available on GitHub.
  • xEdit: Download from Nexus Mods, or GitHub.
  • Bethesda game: xEdit supports Skyrim, Oblivion, Fallout 3, and Fallout New Vegas.

Download and Setup

You can download MXPF from GitHub or Nexus Mods. Once downloaded, you will need to copy the contents of the included Edit Scripts folder into xEdit's Edit Scripts folder. Once you've done that, you're set! You can then use MXPF in any script by putting uses mxpf; after the unit declaration.

User Constants

User constants are constants that are meant to be set by the user. They will vary from machine to machine.

mxDebug boolean
If set to true MXPF will generate debug messages throughout program execution. Use mxSaveDebug, mxPrintDebug or mxEchoDebug to control where debug information is sent.

mxDebugVerbose boolean
If set to true MXPF will generate verbose debug messages. These are messages associated with every record processed by MXPF. Tracking these messages will slow down MXPF execution and may make debug logs extremely large.

mxSaveDebug boolean
If set to true and mxDebug is enabled, debug messages will be saved to a text document in mxpf/logs/mxpf-debug-{{time}}.txt.

mxSaveFailures boolean
If set to true, failure messages will be saved to a text document in mxpf/logs/mxpf-failures-{{time}}.txt.

mxPrintDebug boolean
If set to true and mxDebug is enabled, debug messages will be printed to xEdit's message log when FinalizeMXPF is called.

mxPrintFailures boolean
If set to true, failure messages will be printed to xEdit's message log when FinalizeMXPF is called.

mxEchoDebug boolean
If set to true and mxDebug is enabled, debug messages will be printed to xEdit's message log during execution. Echoing messages slows down script execution.

mxEchoFailures boolean
If set to true, failure messages will be printed to xEdit's message log during execution. Echoing messages slows down script execution.

Developer Constants

Developer constants are constants that shouldn't be changed. These constants will be the same on every machine per version of MXPF, and can be useful in scripts that use MXPF.

mxVersion string
This is a string representing the version of the MXPF file being used. You can check this to make sure that MXPF is of an adequate version for your script.

mxBethesdaSkyrimFiles string
This is a string containing a comma-separated list of the filenames of all Bethesda master files associated with Skyrim.

mxHardcodedDatFiles string
This is a string containing a comma-separated list of the filenames of all hardcoded dat files that xEdit uses to emulate game executables. These files appears as Skyrim.exe, Fallout3.exe, Oblivion.exe, and FalloutNV.exe in xEdit.

Variables

mxLoadMasterRecords boolean
Setting this to true will alter how LoadRecords behaves. When true, LoadRecords will only load records that are masters (i.e. new records in the file they're being processed in, not overrides). This allows you to avoid patching records that override records from files you've excluded or not included.

mxLoadOverrideRecords boolean
Setting this to true will alter how LoadRecords behaves. When true, LoadRecords will only load records that are override records (i.e. not new records). This is the opposite of mxLoadMasterRecords, and is useful if you want to only target override records in a few specific files. Note that using this option and mxLoadMasterRecords at the same time will result in no records being loaded.

mxLoadWinningOverrides boolean
Setting this to true will alter how LoadRecords behaves. When true, LoadRecords will load the winning overrides of records, which makes certain you don't overwrite the changes made to the record that would be present in game without the patch file you're generating. This is extremely important, and should almost always be set to true. This option can be used in conjunction with mxLoadMasterRecords or mxLoadOverrideRecords.

mxSkipPatchedRecords boolean
Setting this to true will have CopyRecordToPatch and CopyRecordsToPatch skip copying records that are already present in mxPatchFile. This is useful if you want to avoid performing duplicate work or patching the same value twice.

mxPatchFile IInterface
Holds the patch file used by MXPF. This is useful to access if you need to change some attributes of the patch file created by your script.

General

InitializeMXPF procedure InitializeMXPF;
Initializes MXPF. You must call this at the beginning of every script before attempting to access any other MXPF function (except DefaultOptionsMXPF).

DefaultOptionsMXPF procedure DefaultOptionsMXPF;
Sets default options in MXPF. Usually you should call this whenever you use MXPF, as it sets the options that most use cases require. This method sets mxLoadMasterRecords, mxLoadWinningOverrides and mxSkipPatchedRecords to true.

FinalizeMXPF procedure FinalizeMXPF;
Finalizes MXPF. You must call this at the end of every script you've used MXPF in. Calling this function frees all memory used by MXPF, cleans the masters of mxPatchFile if it is assigned, and prints/saves debug and failure messages, depending on user constants. 

unit userscript;

uses mxpf;

function Initialize: integer;
begin
  InitializeMXPF;
  DefaultOptionsMXPF;
  { Everything you want to do with MXPF goes here. }
  FinalizeMXPF;
end;

Patch File Selection

PatchFileByAuthor procedure PatchFileByAuthor(author: string);
Selects the MXPF patch file by an author. If no file is found in the user's load order matching the specified author, the method will prompt the user to create one, and set the new file's author to the specified author string.
Once selected, the patch file is stored in the mxPatchFile variable, and is used by MXPF Record Patching methods. Using author based file selection is preferable over filename selection because xEdit does not currently support a means of creating a file with a specified name (the user has to enter the name manually). Author selection also allows the user to have multiple instances of the patch file in their load order without having to manually rename them to avoid conflicts.
NOTE: If there are multiple files with authors matching the specified author, this method will select the one closest to the top of the user's load order. 

PatchFileByAuthor('MXPF ARMO Example');

PatchFileByName procedure PatchFileByName(filename: string);
Selects the MXPF patch file by a filename. If no file is found in the user's load order matching the specified filename, the method will prompt the user to create one. If the created file does not match the desired filename, the script will inform the user that it will not recognize that file as the patch file in the future.
Once selected, the patch file is stored in the mxPatchFile variable, and is used by MXPF Record Patching methods. Using filename based file selection is fairly standard, but selecting by author is better because it is more flexible. 

PatchFileByName('MXPF ARMO Example.esp');

File Selection

SetExclusions procedure SetExclusions(s: string);
Sets the file selection mode to exclusion and loads the comma-separated list of files, s, into the mxFiles stringlist. Affects which files LoadRecords will load records from. 

PatchFileByAuthor('MXPF ARMO Example');
SetExclusions(mxHardcodedDatFiles);
// now we won't load records from any of the hardcoded dat files
LoadRecords('ARMO');

SetInclusions procedure SetInclusions(s: string);
Sets the file selection mode to inclusion and loads the comma-separated list of files, s, into the mxFiles stringlist. Affects which files LoadRecords will load records from. 

PatchFileByAuthor('MXPF ARMO Example');
SetInclusions('Skyrim.esm');
// now we will only load records from Skyrim.esm
LoadRecords('ARMO');

Record Processing

LoadRecords procedure LoadRecords(sig: string);
Loads records from the file selection specified with SetInclusions or SetExclusions. All records matching the input signature, sig will be added to the mxRecords list. Note: Calling this function does not clear the mxRecords list. 

SetExclusions(mxHardcodedDatFiles);
LoadRecords('WOOP'); // loads all WOOP (word of power) records
LoadRecords('OTFT'); // loads all OTFT (outfit) records

LoadChildRecords procedure LoadChildRecords(sig: string);
Loads records from the file selection specified with SetInclusions or SetExclusions. All records matching the input signature, sig will be added to the mxRecords list. You must use this function instead of LoadRecords when loading records from child groups. These are Dialogue Topic Info records (INFO) and any records that are children of Cell or Worldspace records (NAVM, ACHR, REFR, etc.). 

SetExclusions(mxHardcodedDatFiles);
LoadChildRecords('NAVM'); // loads all NAVM (navmesh) records
LoadChildRecords('REFR'); // loads all REFR (object reference) records

GetRecord function GetRecord(i: integer): IInterface;
Accesses records that were loaded through the LoadRecords function. Takes an index, i, to determine which record to load. You should use this function in conjunction with MaxRecordIndex. 

LoadRecords('STAT');
for i := 0 to MaxRecordIndex do begin
  rec := GetRecord(i);
  AddMessage('Found '+Name(rec));
end;

RemoveRecord procedure RemoveRecord(i: integer);
Removes the record at the specified index from the mxRecords list. Use this to apply additional filtering to records you want to patch after calling LoadRecords and before calling CopyRecordsToPatch. NOTE: You should always loop from the maximum index downto 0 when removing elements from the set you're looping through. 

LoadRecords('WEAP');
for i := MaxRecordIndex downto 0 do begin
  rec := GetRecord(i);
  damage := StrToInt(GetElementEditValues(rec, 'DATA\Damage'));
  // remove the weapon if its damage is less than 20
  if damage < 20 then
    RemoveRecord(i);
end;

MaxRecordIndex function MaxRecordIndex: integer;
Gets the number of items in mxRecords minus 1, the maximum index you can call GetRecord with successfully. This is for use when looping through loaded records.

Record Patching

CopyRecordToPatch function CopyRecordToPatch(i: integer): IInterface;
Copies the record at index i in mxRecords to the mxPatchFile. Adds required masters to mxPatchFile if they haven't been added yet. This provides more control than CopyRecordsToPatch. 

LoadRecords('NPC_');
for i := 0 to MaxRecordIndex do begin
  rec := GetRecord(i);
  // instead of using RemoveRecord, we can just copy specific
  // records with CopyRecordToPatch
  if GetElementEditValues(rec, 'ACBS\Flags\Female') = '1' then
    CopyRecordToPatch(i);
end;

CopyRecordsToPatch procedure CopyRecordsToPatch;
Copies all loaded records in mxRecords to the mxPatchFile. Adds required masters to mxPatchFile if they haven't been added yet. Records that fail to copy will be logged to mxFailures. Records that are copied successfully will be added to mxPatchRecords, and will be accessible through GetPatchRecord.

GetPatchRecord function GetPatchRecord(i: integer): IInterface;
Accesses records that were copied through the CopyRecordToPatch and CopyRecordsToPatch functions. Takes an index, i, to determine which record to load. You should use this function in conjunction with MaxPatchRecordIndex. 

LoadRecords('LSCR');
CopyRecordsToPatch;
for i := 0 to MaxPatchRecordIndex do begin
  rec := GetPatchRecord(i);
  AddMessage('Patched '+Name(rec));
end;

MaxPatchRecordIndex function MaxPatchRecordIndex: integer;
Gets the number of items in mxPatchRecords minus 1, the maximum index you can call GetPatchRecord with successfully. This is for use when looping through patch records.

Reporting

PrintMXPFReport procedure PrintMXPFReport;
Prints a message to log the reporting on the number of copy operations performed, the number that were successful, and the number that failed. 

LoadRecords('INGR');
CopyRecordsToPatch;
PrintMXPFReport;
